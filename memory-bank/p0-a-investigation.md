## 2. Root Cause Analysis (Hypothesis)

Based on the symptom ("column does not exist" when psql confirms it does, after `prisma generate` succeeds), the most likely causes are:

### 2.1 Hypothesis A: Stale Prisma engine binary (MOST LIKELY)

The Prisma 5.22.0 engine binary may have cached introspection metadata from an earlier schema state (before `20260330_add_tier_system`). When `prisma generate` regenerates the JS client, it doesn't always invalidate the engine's compiled query plan cache.

**Evidence supporting this:**
- The error says column doesn't exist in DB
- psql confirms column DOES exist
- `prisma generate` completes successfully but doesn't fix it
- This is a known class of issue with Prisma 5.x

### 2.2 Hypothesis B: Process connected to wrong DB (POSSIBLE)

The PM2 process on Contabo may have a stale `DATABASE_URL` env var pointing to a different DB than the one being inspected via psql.

**Evidence supporting this:**
- activeContext.md notes DATABASE_URL is set via SSH tunnel to Contabo
- If the PM2 process was started before the tunnel was re-established, it might point to a stale local DB or a different remote DB
- Recent Contabo migration (per CONTABO_MIGRATION_PLAN.md) could have changed DB endpoints

### 2.3 Hypothesis C: Schema drift from `prisma db pull` (LESS LIKELY)

Someone may have run `prisma db pull` against the live DB at some point, regenerating the schema from DB state. If the DB at that moment didn't have the tier columns yet, the regenerated schema wouldn't include them, and the migration would never apply. But this would have been caught by the existing `git checkout` fix.

### 2.4 Hypothesis D: Stale `dist/` build (POSSIBLE)

If the backend was built (`pnpm build`) before the tier migration was applied, the compiled JS would have an older Prisma client embedded. `pnpm build` after `prisma generate` is needed.

## 4. Contabo Runbook — Executable Fix Steps

**Run these in order on Contabo. Stop after Step 3 if Step 3 confirms columns exist.**

### Step 1 — Snapshot current state

```bash
ssh contabo
cd /opt/neurecore/backend  # or wherever backend lives
pm2 list
pm2 logs neurecore-backend --lines 50 --nostream | tail -30
```

Capture the actual error from logs. Look for lines like:

```
The column `agents.tierAgentPoolId` does not exist in the current database.
```

### Step 2 — Identify which DB the running process uses

```bash
# What DB does the PM2 process think it's using?
pm2 env 0 | grep -E "DATABASE_URL|POSTGRES"
# or
cat /opt/neurecore/backend/.env | grep DATABASE_URL
# or
pm2 show 0 | grep -A2 "env"
```

Then:

```bash
# What DB does psql see?
PGPASSWORD=$(grep PGPASSWORD /opt/neurecore/backend/.env.production 2>/dev/null | cut -d= -f2 | tr -d '"')
psql "$(grep DATABASE_URL /opt/neurecore/backend/.env.production | cut -d= -f2 | tr -d '"')" -c "\dt"
psql "$(grep DATABASE_URL /opt/neurecore/backend/.env.production | cut -d= -f2 | tr -d '"')" -c "\d agents" | grep tierAgentPoolId
psql "$(grep DATABASE_URL /opt/neurecore/backend/.env.production | cut -d= -f2 | tr -d '"')" -c "\d tenants" | grep tierId
```

**If DBs differ between Step 2a and 2b → that's the bug.** Fix env vars and restart PM2. Done.

**If DBs are the same and columns exist → proceed to Step 3.**

### Step 3 — Verify migration status

```bash
cd /opt/neurecore/backend
pnpm prisma migrate status
```

Expected output: "Database schema is up to date" or list of pending migrations.

**If pending migrations show `20260330_add_tier_system` → run `pnpm prisma migrate deploy`** (production-safe; does not lose data).

**If no pending migrations but error persists → proceed to Step 4.**

### Step 4 — Force regenerate Prisma client + clear all caches

```bash
cd /opt/neurecore/backend

# 4a. Stop backend
pm2 stop neurecore-backend

# 4b. Nuke all Prisma caches
rm -rf node_modules/.prisma
rm -rf node_modules/@prisma/client/runtime
rm -rf node_modules/.cache
rm -rf dist/

# 4c. Reinstall Prisma client fresh
pnpm install --force @prisma/client@5.22.0
pnpm install --force prisma@5.22.0

# 4d. Regenerate
pnpm prisma generate

# 4e. Rebuild backend
pnpm build

# 4f. Restart
pm2 restart neurecore-backend

# 4g. Verify
pm2 logs neurecore-backend --lines 30 --nostream | tail -20
curl -H "Authorization: Bearer $TOKEN" http://localhost:3000/api/v1/agents?limit=1
```

**If still failing → Step 5.**

### Step 5 — Nuke and rebuild engine binary

```bash
cd /opt/neurecore/backend
pm2 stop neurecore-backend

# 5a. Find the engine binary
find node_modules/@prisma/engines -name "*.node" -o -name "*.js" | head -10
ls -la node_modules/@prisma/engines/

# 5b. Delete and reinstall
rm -rf node_modules/@prisma
pnpm install

# 5c. Verify engine version
node -e "const e = require('@prisma/engines'); console.log(Object.keys(e))"
node -e "console.log(require('@prisma/client/package.json').version)"  # should be 5.22.0

# 5d. Regenerate + build + restart
pnpm prisma generate
pnpm build
pm2 restart neurecore-backend
sleep 5
curl -H "Authorization: Bearer $TOKEN" http://localhost:3000/api/v1/agents?limit=1
```

**If still failing → Step 6 (escalation).**

### Step 6 — Escalation

If after Step 5 the error persists, the issue is likely:

- DB connection actually goes to a different DB than inspected
- Migration was applied but to wrong DB
- TierAgentPool / Tier tables don't actually exist (only the columns were added)

Diagnose:

```bash
# 6a. Are the TIER tables present?
psql "$DATABASE_URL" -c "\dt" | grep -i tier
# Expected: tiers, tier_agent_pools

# 6b. Is there a _prisma_migrations table?
psql "$DATABASE_URL" -c "\d _prisma_migrations"
# Expected: contains rows for all applied migrations

# 6c. What migrations does Prisma think were applied?
psql "$DATABASE_URL" -c "SELECT migration_name, finished_at FROM _prisma_migrations ORDER BY started_at;"
# Expected: includes "20260330_add_tier_system"
```

**If `_prisma_migrations` is missing `20260330_add_tier_system` → run `pnpm prisma migrate deploy`** (production-safe; idempotent).

**If migration IS recorded as applied but columns still missing → manually re-apply:**

```bash
PGPASSWORD=... psql "$DATABASE_URL" < prisma/migrations/20260330_add_tier_system/migration.sql
```

**If after all this it still fails → open GitHub issue on `prisma/prisma` with:**

- `prisma --version`
- `node --version`
- Full error trace from PM2 logs
- Output of `psql "\dt"` and `psql "\d agents"`
- Output of `pnpm prisma migrate status`
